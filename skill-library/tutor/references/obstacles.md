# Universal Learning Obstacles

A wrong answer is rarely random — it usually comes from a consistent but faulty pattern the learner is applying. The tutor's job is to surface that pattern and break it, not just supply the correct answer (which leaves the faulty pattern intact to misfire next time).

The most reliable fix is almost always **a case the learner evaluates themselves** — being told you're wrong changes less than watching your own rule produce something you know is false.

## How to use this file

Reach for it when an error looks *conceptual* (a consistent wrong pattern) rather than a *slip* (a fatigue error, a typo, a momentary lapse). Slips need "check that line again"; conceptual errors need the treatment below.

**Slip vs. misconception:** A learner who drops a sign once likely slipped. A learner who *always* drops it the same way, or who applies a wrong rule consistently across different problems, has a misconception. The response is different: slips need a nudge; misconceptions need a contradiction the learner generates.

---

## Overgeneralization — extending a rule past its valid domain

**Pattern:** A rule learned in context A gets applied blindly in context B where it doesn't hold. Seductive because the rule *does* work — just not everywhere.

**Examples across domains:**
- Math: "Multiplication makes bigger" (true for integers > 1, false for fractions).
- Writing: "Never start a sentence with 'And'" (a style rule overapplied as a grammar rule).
- Programming: "Functions always return a value" (true in math, not always in code).
- Music: "The root note is always in the bass" (common in simple harmony, false in inversions).

**Fix:** Have them apply their rule to a case at the boundary. Don't tell them it's wrong — ask "does your rule hold here?" and let the contradiction emerge from their own computation. Then name the actual valid domain: "That rule works when X; here we have Y."

---

## Undergeneralization — treating a specific case as if it doesn't generalize

**Pattern:** The learner learned X in one specific context and doesn't see it applies broadly.

**Examples:**
- Learned to solve quadratic equations by factoring but doesn't recognize the quadratic formula as the same operation in disguise.
- Knows to cite sources in essays but doesn't apply the same standard to informal arguments.
- Can debug their own code but doesn't apply the same "trace the assumption" method when reading someone else's.

**Fix:** Point to the structural similarity: "What does this situation have in common with the one where you solved X?" Make the pattern explicit rather than hoping they'll spot it.

---

## False analogy — reasoning from an analogy that breaks in a key way

**Pattern:** The learner has a mental model (often a good one at first) that they extend into territory where the analogy no longer holds.

**Examples:**
- "Electricity flows like water" is a useful model until someone asks about resistance in parallel circuits.
- "A word's meaning is what the dictionary says" breaks when studying how meaning is use-dependent.
- "Natural = safe" transfers from food intuitions to medicine dosing, where it fails.

**Fix:** First acknowledge *where the analogy works* — don't dismiss the mental model entirely, or you destroy a useful scaffold. Then surface the specific place it breaks: "That model is useful up to this point — where does it start to predict the wrong thing?" Have them identify the crack themselves.

---

## Premature closure — thinking understanding is complete before it is

**Pattern:** The learner reaches a plateau that feels like mastery. They can follow explanations, reproduce examples they've seen, and pass surface-level checks. They stop there.

**Signs:** Can solve problems that look like the examples but fail on any twist. Can recite the rule but not explain why it's true. Feels confused when asked "when would this fail?"

**Fix:** Apply the Bloom's hierarchy: if they're at "understand," push to "apply" with a new context; if at "apply," push to "analyze" with a why-question or boundary case. The twist problem is the primary tool. See `pedagogy.md` → Bloom's Taxonomy.

## Passive recognition — "I followed it" mistaken for "I can do it"

**Pattern:** The learner follows a worked example and believes they've learned it. Recognition and recall feel identical in the moment; they are not.

**Fix:** Close the loop with active retrieval rather than passive confirmation. "Without scrolling up, walk me through the two key steps." If they can't, the break was earlier than the explanation. Don't re-explain — go back further and rediagnose.

---

## Transfer failure — knowing in context A but not applying in context B

**Pattern:** The learner has genuinely understood something but in a narrow, context-locked way. The knowledge doesn't activate when the surface features change.

**Examples:**
- Understands probability in card games, doesn't apply the same reasoning to medical testing.
- Knows how to structure an argument in an essay, doesn't apply it to verbal debate.
- Learned a design pattern in one programming language, doesn't see it in another.

**Fix:** Name the abstract principle explicitly. "What you did there was X — let's call it X. Now here's a completely different-looking problem — where do you see X in here?" Force the generalization to happen consciously, not by accident.

---

## Motivated reasoning — reasoning backward from a desired conclusion

**Pattern:** The learner reaches a conclusion first (often because it's comfortable or familiar) and then constructs justification for it, rather than following the evidence.

**Signs:** Arguments that work only in one direction; resistance to counterexamples; "yes, but" responses to clear contradictions.

**Fix:** Separate the two moves explicitly: "Let's set aside whether this is true for a moment — does the reasoning itself hold?" Then ask them to steelman the opposite: "What's the strongest argument against your position?" This isn't about being adversarial — it's about building the habit of testing reasoning independently of its conclusion.

---

## The general principle

Whatever the domain, the pattern repeats: identify the consistent wrong rule, then engineer a moment where the learner *applies their own rule and watches it produce something they know is false*. That self-generated contradiction does more than any correction you could state. Supply the correct model only after the faulty one has been dislodged — otherwise it lands on top of a belief that's still standing.
