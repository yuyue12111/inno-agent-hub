# Step 3 — 问题转搜索词 + 真实检索 (Question → Query → Retrieve)

> 作用：把抽象问题变成可搜索的查询，去拿真实证据。这是 research 与 persona-prompting 的分界线。

## 这是研究与 persona-prompting 的分界线。绝不能跳过。

## 3a. 问题转搜索词

```
讨论的主题：{TOPIC}
要回答的问题：{QUESTION}

你要用搜索引擎回答这个问题，会在搜索框输入什么？
列出 1-3 条搜索词：
- 查询 1
- 查询 2
- 查询 3
```

## 3b. 真实检索（用工具，不是用记忆）

> 不同环境（Claude Code / Codex / 其他 agent）拥有的检索能力各不相同。
> 检索能力已在 **Step 0（检索能力探测）** 通过**实际遍历真实工具/skill 清单**得到（不是凭记忆），这里直接复用那份清单，不重新发现。
> 若 Step 0 被跳过或清单可疑，回到 Step 0 先把环境工具**完整遍历一遍**再检索。

### 第一步：用 Step 0 的清单尽可能多地并行调用（多源 = 覆盖更广）

- 对每条搜索词，**调用尽可能多的可用搜索工具**（不要只用一个）。多个独立来源能交叉验证、扩大覆盖、抵消单一引擎的偏见。
- 没有依赖关系的检索调用**并行发起**。
- **某个搜索工具返回空 / 失效时绝不停下**：立刻换下一个可用搜索工具；若搜索类工具普遍失效，直接对已知权威域名（PubMed、arXiv、官方站、维基等）用抓取工具取正文兜底，或派发检索子 agent。这是真实高频故障，必须有预案。
- 对返回结果里最相关的 2-3 个 URL，用可用的抓取工具取正文，提取关键 snippet。
- 合并所有工具的结果，去重后进引用表。

### 第二步：降级阶梯（前一级不可用才走下一级）

```
1. 内置检索/抓取工具 —— 有就尽量全用
        ↓ 都没有
2. 检索类 MCP / 检索类 skill —— 有就尽量全用
        ↓ 都没有
3. 派发有检索能力的子 agent（researcher / content-research 等）代为检索，取回结果
        ↓ 无法派发
4. 浏览器自动化兜底：驱动浏览器打开多个搜索引擎页面，
   尽可能多地搜索并从结果页/目标页抓取信息
        ↓ 连浏览器能力都没有
5. 明确告知用户「当前环境无任何可用检索能力，无法做接地研究」，
   不要用模型记忆冒充检索结果。
```

> 提示：很多环境把检索工具授权给了专门的研究子 agent，而非主线程。
> 主 agent 直接调不到搜索时，优先考虑「派发子 agent」这条路，别急着退到浏览器。

> **底线**：只要环境里存在任何一种联网能力，就必须真正联网取证，宁可多调几个工具，也不要跳过检索。这是 research 与 persona-prompting 的分界线。

### 第三步：建立全局引用表（贯穿整个研究）

```
[1] {title} — {url}    (来源工具: <实际取回它的工具名>)
    snippet: "<取回的关键原文片段>"
[2] {title} — {url}    (来源工具: <实际取回它的工具名>)
    ...
```

## 规则

- 同一来源复用同一编号，不重复编号。
- 优先取**一手/权威来源**（官方文档、论文、原始数据），其次主流媒体，最后才是二手解读。
- 记录每条来源是哪个工具取回的（便于判断可靠性、排查偏见）。
- 若所有可用工具都检索不到有用结果，记录"该问题无可靠来源"，进入 Step 4 时据实拒答。
- 去重：相同 URL 只保留一条（即使来自不同工具）。

## 为什么这样设计

检索（Retrieval）是整套方法的命根子：先检索、再作答，检索不到就明确拒答，绝不用记忆补全。这一步把 LLM 从"凭记忆答"变成"带证据答"，是它敢叫 research 的唯一底气。少了它，多视角也只是同一个模型的同一套盲区在自说自话。
